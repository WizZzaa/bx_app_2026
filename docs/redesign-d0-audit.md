# D0-аудит полного редизайна BX

Дата: 20 июля 2026 года
Статус: аудит кода, зависимостей, маршрутов и путей хранения завершён; продуктовый код и зависимости не изменялись.

Security P0 уточнён отдельным read-only отчётом: [`security-p0-posture-2026-07-20.md`](./security-p0-posture-2026-07-20.md).

## 1. Результат

Редизайн нельзя выполнять одной заменой CSS или подключением Motion в одном пакете. Репозиторий содержит четыре технически самостоятельных frontend-контура и семь пользовательских представлений:

1. Electron/Windows.
2. Авторизованный BX Web на широком экране.
3. Авторизованный BX Web на мобильном.
4. Публичный сайт на широком экране.
5. Публичный сайт на мобильном.
6. Отдельный `ip-web`/PWA для ИП.
7. Административная панель.

Desktop-код переиспользуется Web-сборкой через Vite alias, но пакеты, lock-файлы, версии Vite и Tailwind разделены. Пользовательские данные находятся одновременно в Supabase, IndexedDB/Dexie, `localStorage`, Electron `safeStorage` и Cache Storage PWA. До начала D1 запрещены массовая замена оболочки, переименование ключей хранения и установка Motion без выполнения ворот из § 10.

### Motion for React

- `motion` и `framer-motion` сейчас не установлены.
- Проверенная актуальная версия пакета `motion`: `12.42.2`.
- Peer range пакета допускает React 18 и React 19.
- Unpacked package size — около 682 КБ; это не равно финальному bundle impact, который измеряется отдельным пробным branch/build.
- Установка только в `web` недостаточна: Electron собирается из `app`, а Web импортирует renderer из `app`, разрешая зависимости через собственный `node_modules`.
- До установки выбирается один контракт поставки: синхронная зависимость в `app` и `web` либо общий workspace/package. Дублирование разных версий запрещено.

### Security audit зависимостей

- `app`: 1 High vulnerability в прямой зависимости `xlsx` — Prototype Pollution и ReDoS; `npm audit` не предлагает автоматическое исправление.
- `web`: тот же 1 High vulnerability в прямой зависимости `xlsx` без автоматического исправления.
- `ip`: известных production-уязвимостей по `npm audit` нет.
- Добавление Motion не должно маскировать этот риск. До D1 требуется отдельное решение по безопасному XLSX-парсеру, изоляции обработки недоверенных файлов или замене источника пакета.

## 5. Пользовательские данные и запрет на потерю

### 5.1. Локальные данные Desktop/Web

`BusinessBxDatabase` (Dexie, версия 4) содержит:

- `transactions`;
- `employees`;
- `counterparties`;
- `conflicts`;
- `templates`;
- `exchange_rates`.

Критические категории `localStorage`:

- доступ и устройство: установка, Telegram migration, PIN hash/attempts, idle lock;
- рабочий контекст: активная компания, sidebar, тема, плотность, масштаб, workspace modes;
- несинхронизированные данные: `bx_sync_queue_v1`, локальные AI-чаты и сообщения;
- документы и работа: quick notes, support draft, checklist templates, calculator/INN/number histories;
- кеши: события, задачи, boards/cards, знания, сервисы, курсы и справочники;
- Бикс: состояние, настройки, инвентарь и очередь экономики;
- ЭЦП: `bx_ecp_keys_enc` в Electron и открытый fallback `bx_ecp_keys` в Web.

Есть динамические ключи с ID сущности, например локальные AI-сообщения, карточки, сотрудники, транзакции, company details и favorites. Простого списка статических строк недостаточно для миграции.

### 5.2. Облачные данные

Renderer напрямую работает как минимум с компаниями, профилями, участниками, задачами, событиями, досками и карточками, транзакциями, документами, AI-чатами, обращениями, уведомлениями, тарифами, платежами, контентом, справочниками, feature flags и audit/error/widget events.

Миграции 19 июля включают RLS, revoke для `anon`/`authenticated`, `security definer` с заданным `search_path`, защищённые устройства, Telegram identity/recovery, согласия, тарифы/лимиты, двойное подтверждение админских операций и редакционные workflow. Редизайн не должен обходить эти RPC прямым CRUD ради удобства нового UI.

### 5.3. ИП Web/PWA

В `localStorage` без шифрования сохраняются:

- `ip_profile`: ФИО, ИНН/ПИНФЛ, телефон, адрес, банк, МФО и расчётный счёт;
- `ip_reminders`;
- `ip_calc_history`;
- `ip_hidden_presets`;
- локальные AI-сообщения.

Это P0-расхождение безопасности: банковские и идентификационные данные нельзя переносить или переименовывать «по пути» редизайна. Сначала утверждаются шифрование/облачная миграция, экспорт, восстановление и политика очистки. Service worker `ip-shell-v1` использует network-first и Cache Storage; смена shell/cache name требует контролируемого обновления, чтобы PWA не застряла на смешанной версии.

### 5.4. Контракт миграции данных

До изменения любого ключа, store, таблицы или URL обязательно:

1. Составить версионированную таблицу `старое → новое`.
2. Снять обезличенные counts и резервную копию целевого окружения.
3. Добавить чтение старого и нового формата, затем запись нового.
4. Проверить повторный запуск миграции — он должен быть идемпотентным.
5. Проверить offline queue, conflict records, PWA cache и Supabase session.
6. Оставить rollback/dual-read минимум на один стабильный выпуск.
7. Удалять старый формат только после подтверждённой телеметрии и отдельного решения.

## 6. Дизайн-система и CSS

### 6.1. Текущее состояние

- `app` имеет основной набор `--bx-*` в `globals.css`, но Web хранит почти полную копию в `web/src/index.css`.
- Web добавляет отдельные локальные токены лендинга (`--acid`, `--ink`, `--paper`, `--card`) и админки (`--admin-*`).
- `ip` использует независимые `--ip-*` токены и emerald-акцент, не связанный с канонической лавандовой системой BX.
- Найдено 537 вхождений raw hex в App, 299 в Web и 13 в IP. Часть относится к данным/иллюстрациям, поэтому механическая замена запрещена; нужен семантический реестр.
- Найдено 7 Tailwind arbitrary color-классов в App и 10 в Web.
- `landing-v2.css` и `public-pages.css` минифицированы в несколько очень длинных строк, что затрудняет ревью, page overrides и визуальную регрессию.

### 6.2. Motion-аудит

| Контур | `@keyframes` | `animation:` | `transition:` | reduced-motion references |
|---|---:|---:|---:|---:|
| App | 3 | 4 | 10 | 4 |
| Web | 16 | 39 | 19 | 3 |
| IP | 0 | 0 | 0 | 0 |

Web содержит продолжительные декоративные `marquee`, `blob-movement`, `float` и похожие эффекты. Они не переносятся в Motion автоматически. Сначала каждое движение получает назначение: navigation, overlay, feedback, loading, shared-layout или decorative/remove.

### 6.3. UI UX Pro Max — обязательные правила D1

- Bento отражает приоритет, а не превращает каждый блок в карточку.
- Один главный CTA на экран.
- 375/390, 768, 1024 и 1440 px плюс landscape.
- Touch target не меньше 44 × 44 px и расстояние между соседними целями не меньше 8 px.
- Основной мобильный текст не меньше 16 px; отсутствие horizontal scroll.
- Контраст основного текста не ниже 4.5:1; цвет не является единственным носителем статуса.
- Motion: 150–300 мс, сложный переход до 400 мс, exit короче enter, преимущественно transform/opacity.
- Не более двух заметных motion-акцентов на экран; бесконечное движение допустимо только для индикатора загрузки.
- `prefers-reduced-motion`, клавиатура, предсказуемый Back и сохранение scroll/filter state обязательны.

## 7. Производительность до редизайна

Все production-сборки проходят, но уже превышают рекомендуемый chunk threshold:

| Контур | Главный JS | gzip | CSS | gzip |
|---|---:|---:|---:|---:|
| App | 5 243.58 КБ | 1 719.54 КБ | 138.67 КБ | 22.95 КБ |
| Web | 5 073.69 КБ | 1 501.69 КБ | 302.93 КБ | 43.08 КБ |
| IP | 601.31 КБ | 172.51 КБ | 21.34 КБ | 5.10 КБ |

Дополнительно App/Web включают несколько PNG Бикса размером 1.4–2.0 МБ каждый и PDF worker около 2.2 МБ. Основные Desktop-маршруты импортируются статически; route-level splitting есть только у панелей админки. До Motion требуются lazy routes, выделение тяжёлых document/PDF/OCR/XLSX модулей и бюджет initial bundle.

Предварительный бюджет для D1, который ещё должен быть утверждён:

- Motion не увеличивает gzip initial JS более чем на 25 КБ для маршрута, который его использует.
- Публичный landing не загружает Desktop renderer, PDF/OCR/XLSX и ассеты Бикса до явной необходимости.
- CLS < 0.1; INP и LCP фиксируются до/после на одном устройстве и сети.

## 9. Критические расхождения

### P0 — до установки Motion и изменения оболочки

1. Токен Telegram-бота, ранее переданный через чат, считается раскрытым: до production его нужно перевыпустить в BotFather, заменить только в secret storage окружений и отозвать прежнее значение.
2. High vulnerability в `xlsx` у App/Web без автоматического fix.
3. Незашифрованные персональные и банковские данные `ip_profile` в localStorage.
4. Нет единого package/workspace-контракта для зависимости, используемой общим renderer.
5. Нет версионированного реестра localStorage/IndexedDB/PWA/Supabase migrations для UI-релиза.
6. Большой незакоммиченный набор пользовательских миграций и Web-правок нельзя переписывать или откатывать в рамках редизайна.

### P1 — до массового переноса страниц

1. App/Web дублируют токены, а landing/admin создают локальные палитры.
2. `ip-web` отсутствовал в первоначальном перечне поверхностей редизайна.
3. Публичный сайт не реализует канонические RU/O‘Z URL.
4. Админские панели не имеют URL/deep link и теряют активную панель при reload.
5. Web загружает общий многомегабайтный renderer; публичная и авторизованная поверхности недостаточно изолированы.
6. Мобильный лендинг использует масштабирование широкого dashboard-макета вместо самостоятельной композиции.
7. В IP нет reduced-motion, high-contrast, dark/system theme и формализованных responsive tests.

## 10. Ворота перехода к D1

D1 можно начать только в следующем порядке:

1. **Security decision:** ротировать раскрытый Telegram-токен и зарегистрировать решение по `xlsx` и `ip_profile`.
2. **Data contract:** добавить machine-readable реестр хранилищ и тесты dual-read до изменения ключей.
3. **Dependency architecture:** выбрать поставку Motion для Electron и Web, зафиксировать одну версию и измеримый bundle budget.
4. **Quality baseline:** сохранить зелёные App build/typecheck/tests; для Web исправить lint либо утвердить временный baseline без новых нарушений; добавить typecheck/lint минимум для IP.
5. **Design foundation:** создать `design-system/MASTER.md`, page overrides и семантические токены без массовой замены экранов.
6. **Vertical slice:** Dashboard + навигация + один overlay в App/Web/Mobile Web, без изменения моделей данных.
7. **Canary:** визуальная, клавиатурная, mobile, reduced-motion и data-retention приёмка; только после неё перенос остальных маршрутов.

## 11. Следующий безопасный пакет работ

Следующий пакет не меняет бизнес-данные:

1. Ротировать раскрытый Telegram-токен; оформить решения P0 по `xlsx`, `ip_profile` и способу поставки Motion.
